Calcul parallèle — ENSAI 3A

Notes de conception du cours (fichier gitignored, à usage interne / assistant).

Contexte et public

  • Cours en 4 modules (on dit « modules », pas « séances ») :
    1. compilation & spécialisation par les types ;
    2. multiple dispatch vs POO Python (point de langage, sans parallélisme) ;
    3. threads & concurrence / race conditions ;
    4. hiérarchie mémoire, batching & GPU.
  • Dispatch et threads sont deux modules distincts (ils n’ont rien à voir : l’un est un point de langage, l’autre du parallélisme). Découpage demandé explicitement.
  • Fil rouge du cours : l’estimation de π par Monte-Carlo (séquentiel → threads → GPU).
  • Étudiants ENSAI 3A : à l’aise en Python, débutants en Julia.
  • Angle directeur du cours : Python comme point de comparaison, Julia comme banc d’essai. La performance ne vient pas du langage « magique » mais de comprendre ce que fait la machine (coût par opération, partage, déplacement des données). Les démos chocs reposent sur la comparaison de chiffres entre Python et Julia (ex. boucle Python ~1 s vs boucle Julia ~ms).

Fichiers actuels

pluto/ contient les notebooks Pluto (Julia), nommés sans « séance », préfixés par leur ordre : - 1_compilation_et_types.jl - 2_dispatch_vs_poo.jl - 3_threads_et_concurrence.jl - 4_memoire_et_gpu.jl

Statut : ce sont des corrigés / supports de présentation, pas des activités étudiantes (toutes les solutions sont visibles, et c’est du Julia pur). Les renommer/garder comme tels.

python/ contient les analogues Python (Colab) des 4 notebooks Julia — les vrais benchmarks de la piste Python (cf. décision plus bas) : - 1_compilation_et_types_python.ipynb — boucle pure vs numpy vs numba njit ; dis (bytecode) ; - 2_dispatch_vs_poo_python.ipynb — POO, singledispatch, multipledispatch (conceptuel, pas de bench) ; - 3_threads_et_concurrence_python.ipynb — GIL (threads vs process), race condition, MC π en process ; - 4_memoire_et_gpu_python.ipynb — localité C/F-order, intensité, batching, PyTorch CPU vs CUDA. Générés par /tmp/build_py_notebooks.py (script jetable : éditer là-bas puis relancer si besoin). Points délicats : numba/multipledispatch via !pip install (Colab) ; la race condition utilise time.sleep(0) pour exposer l’entrelacement (le GIL le masque sinon — le bug reste réel) ; le GPU est gardé par torch.cuda.is_available().

slides/cours.qmd : un seul deck reveal.js (voir section Slides).

Statut Python : corrigés/démo de la piste Python. Les activités (squelettes à trous) restent à dériver de ces corrigés.

Renvois croisés entre notebooks : par thème, pas par numéro (« le module sur les threads », « vu avec la compilation »…) — robuste si l’ordre change.

Stratégie pédagogique de format (décision)

Pluto = Julia uniquement + réactif (montre la solution immédiatement). Parfait pour présenter en direct et comme corrigé, inadapté comme TP pour ce public. On sépare donc trois rôles et on découple la langue de l’activité :

  1. Corrigés / démo prof → les Pluto actuels (réactifs, projetables). Non distribués comme TP.
  2. Activités étudiantes = deux pistes parallèles sur la même tâche, au choix :
    • Piste Python (tronc commun, pour tous) : Jupyter/Colab, zéro install. Ancre familière. time/timeit, numpy, éventuellement numba.
    • Piste Julia (approfondissement, optionnelle) : squelette à trous (# À COMPLÉTER), pour voir @btime, @code_warntype, @threads, le GPU. La comparaison des chiffres entre les deux pistes (mise en commun en classe) est le moteur pédagogique. Le tronc commun Python garantit que personne ne décroche ; la piste Julia récompense la curiosité.
  3. Sujet / énoncé → document Quarto (cohérent avec le site epilliat.github.io) ou PDF : énoncé + questions conceptuelles langage-agnostiques, blocs solution repliés/masqués.

Format recommandé pour les activités : Jupyter / Colab (Python et Julia via IJulia) — les étudiants connaissent Jupyter, et Colab supprime la friction d’installation (le vrai obstacle étant l’install de Julia, pas Pluto en soi).

Différenciation par module : définir un tronc commun (faisable en Python par tous) + extensions Julia (pour approfondir). Ex. Monte-Carlo π : tous chronomètrent Python pur vs numpy ; ceux qui vont plus loin écrivent la boucle Julia + @btime, puis les threads.

Benchmarks Python/numpy/PyTorch — décision

  • Les notebooks Julia (Pluto) NE font PAS d’appel Python. Les comparaisons Python y restent en prose (chiffres cités : boucle Python ~1 s, numpy.sum ~10 ms, etc.). Pas de PyCall / PythonCall / torch dans les Pluto — ça alourdit (CondaPkg, env conda téléchargé au 1er run ; torch = centaines de Mo + risque CUDA), c’est fragile pour un corrigé, et le benchmark honnête imposerait de chronométrer côté Python (timeit) pour éviter la frontière FFI.
  • Les vrais benchmarks Python/numpy/PyTorch vivent dans la piste Colab (natif, zéro install ; torch+GPU trivial sur Colab). C’est là qu’on mesure réellement, et qu’on compare aux chiffres Julia en mise en commun. Cohérent avec le découplage des langues.
  • Si un jour on veut une démo live côté Julia : appendice optionnel PythonCall (pas PyCall), chronométré côté Python — mais ce n’est pas le défaut retenu.

Convention de nommage suggérée (si on crée sujets/activités)

Préfixe numérique d’ordre, thème, jamais « séance » : - Sujet : N_<theme>_sujet.{qmd,ipynb} - Activité Python : N_<theme>_activite_python.ipynb - Activité Julia : N_<theme>_activite_julia.{ipynb,jl} - Corrigé : les Pluto N_<theme>.jl (existants).

Pièges Pluto rencontrés (à appliquer à tout notebook Pluto ici)

  • Une seule expression par cellule. Plusieurs instructions au niveau racine → erreur « Plusieurs expressions dans une même cellule ». Corriger en entourant de begin … end (le const ... est autorisé dans un begin), ou en scindant en cellules séparées (auquel cas ajouter un UUID # ╔═╡ ... par cellule et mettre à jour la section # ╔═╡ Cell order:).
  • with_terminal() (PlutoUI) est cassé dans l’environnement courant : UndefVarError: value not defined in PlutoUI.TerminalNotebook. Le remplacer par let … end : Pluto capture déjà stdout automatiquement, donc println et les sorties de @btime s’affichent sous la cellule. (Déjà appliqué aux 4 modules.)
  • Exemples volontairement buggés à ne pas « corriger » : la somme_fausse / pi_faux (race condition pédagogique), l’usage de threadid() pour des accumulateurs locaux (correct avec l’ordonnancement :static par défaut de @threads).
